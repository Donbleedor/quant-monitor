# Mahjong V3 UI 实装全面验收报告

日期：2026-06-12

结论：`NEEDS_FIX_BEFORE_FINAL_ACCEPTANCE`

## 2. 总体判断

当前版本已完成资源路径接入、9:20 外壳、基础页面和操作事件，但还不能作为最终视觉验收版。最严重的问题不是美术精细度，而是资源使用方式改变了信息层级和游玩判断：

- 被上层遮挡的牌被显示成紫边黑色空牌，玩家无法直观看出完整牌阵。
- 顶部 HUD 同时叠加完整框、中心板和独立按钮壳，出现重复圆槽和过度装饰。
- 底部托盘自带三个圆槽，又叠加完整圆形按钮壳和文字，造成按钮过重、标签拥挤。
- 模态遮罩是一张不透明全屏纹理，暂停时完全看不到原游戏现场。
- 菜单品牌和主按钮的实际可见尺寸明显小于效果图。

## 3. P0 阻塞问题

### P0-01 覆盖牌状态错误，破坏牌阵可读性

证据：

- `2.png`、`4.png` 中大量下层牌表现为紫边黑绿网格空框。
- 目标效果图中，下层牌仍保留象牙牌体和符号，只通过亮度和饱和度区分层级。
- `style.css:339-345` 将 `state_tile_covered_overlay.png` 以 100% 尺寸完全覆盖牌面。
- 该资源本身是深色网格，并带明显紫色描边。

影响：

- 玩家难以预判下层有哪些牌。
- 牌阵看起来像大量空槽，而不是层叠实体牌。
- 紫色描边不属于深绿、古金、象牙、陶橙的既定色板。

建议：

- 保留象牙牌体和图腾可见。
- covered 状态使用亮度约 `0.42-0.55`、饱和度约 `0.55-0.75`。
- 覆盖层只能是轻微深绿薄雾，不能是不透明替换牌。
- 重新处理或停止使用当前 `state_tile_covered_overlay.png`。

### P0-02 当前资源仍有粉色/紫色抠图残留

人工确认存在明显残留：

- `assets/menu/menu_vignette_bottom.png`
- `assets/modal/modal_panel_large.png`
- `assets/modal/modal_panel_standard.png`
- `assets/modal/icon_close.png`
- `assets/modal/level_node_open.png`
- `assets/modal/btn_modal_secondary.png`
- `assets/modal/toggle_track.png`

自动颜色抽查还提示同批 modal 图标、关卡节点、按钮和 toggle 可能具有同源边缘污染，需要整批复核，不能只修截图里最明显的几张。

影响：

- 在深绿色背景上形成高显眼度粉边。
- 直接违背此前“粉色抠干净”的交付标准。

建议：

- modal P0 与菜单装饰层重新做白底、黑底、棋盘格三底色检查。
- 边缘不得通过简单扩大透明阈值硬切，避免金边被吃掉。
- 对半透明边缘做去色溢出，而不只是把纯 `#FF00FF` 设为透明。

### P0-03 主菜单构图与批准效果图差异过大

当前：

- Logo 实际可见宽度约为屏幕三分之一，挤在顶部。
- 中间出现大面积无功能空区。
- Play 和二级入口被推到屏幕下部。

目标：

- Logo 是第一视觉中心，实际可见宽度约占屏幕 `72%-80%`。
- Play 位于屏幕约 `54%-66%` 高度区间。
- Level/Daily 位于约 `70%-84%` 高度区间。

根因：

- `.menu-logo-wrap` 只有 `70%` 宽，而 Logo PNG 在自身 `760x420` 画布中的可见宽度约 `56%`。
- `.menu-actions` 使用 `margin-top:auto` 和 `margin-bottom:18%`，没有按批准构图定位。
- Play PNG 的实际可见宽度约为自身画布 `74%`，再乘 CSS `78%` 后，主体只占屏幕约 `58%`。

### P0-04 顶部 HUD 重复叠加完整组件

当前结构同时使用：

- `hud_top_frame.png`
- `hud_center_plate.png`
- `hud_btn_home_bg.png`
- `hud_btn_pause_bg.png`

其中 `hud_top_frame.png` 已经带中心面板和两端圆槽，`hud_center_plate.png` 又带一套两端圆槽。截图中形成四个空圆槽，视觉明显比效果图复杂。

建议：

- 保留 `hud_top_frame.png` 作为唯一主框。
- 移除运行时的 `hud_center_plate.png` 叠层。
- 把 mask 和 timer 图标放入主框两侧圆槽，把关卡和时间文本放在中间长板。
- Home/Pause 独立方形按钮继续放在最外侧。

### P0-05 底部工具栏层级过重且文字拥挤

当前：

- `tool_tray_panel.png` 已自带三个圆槽。
- 每个槽上又叠加 `tool_btn_shell.png`。
- 图标下继续叠加 `HINT/UNDO/MIX`。
- 截图中文字靠近底边并与图标争夺空间。

建议：

- 游戏内底栏使用托盘自带圆槽，直接放置三个图标。
- `tool_btn_shell.png` 保留给菜单二级入口或独立按钮，不在托盘内重复使用。
- 去掉可见文字，只保留 `aria-label`。
- 次数徽章附着在圆槽右上，Unlimited 工具隐藏徽章或统一显示 `∞`。

### P0-06 模态遮罩完全替换游戏现场

`modal_overlay_dark.png` 是一张不透明的 `1080x2400` 深色纹理。当前 CSS 将其作为第一层背景：

```css
background: url('assets/modal/modal_overlay_dark.png') center/cover no-repeat,
            rgba(7,18,13,0.85);
```

图片本身没有透明度，因此下面的 rgba 和 gameplay 都不可见。暂停截图变成“新背景 + 弹窗”，失去暂停应有的上下文。

建议：

- 游戏现场保持可见。
- 使用 `rgba(7,18,13,0.62-0.72)` 作为主遮罩。
- 如必须保留纹理，放到伪元素并单独设置 `opacity:0.15-0.25`。
- 低端机优先不用 `backdrop-filter`，或把 blur 降为可选增强。

### P0-07 弹窗按钮文字颜色与底图明暗相反

- Primary 按钮底图是深绿色，当前文字为深色，低对比。
- Secondary 按钮底图是象牙色，当前文字为象牙色，低对比。

对应 `style.css:619-627`。

建议：

- Primary：象牙或亮金文字。
- Secondary：深棕或深绿文字。
- 图标和文字都按实际底图检查，不按“primary/secondary”名称猜颜色。

### P0-08 每日挑战标题未本地化到最终 UI 规则

`main.js:113` 仍传入 `每日挑战`。截图中标题换成两行并与 HUD 装饰重叠。

建议：

- 显示为单行 `DAILY`。
- 普通数字关卡显示 `LV. 2`。
- HUD 文本容器设置 `white-space:nowrap` 和明确的最大宽度。

### P0-09 Home 图标与实际点击结果不一致

- `main.js:133`：Home 按钮调用 `pauseAndShowMenu()`。
- `main.js:303-305`：该函数实际只调用 `pauseGame()`。
- 右侧 Pause 也调用同一个暂停流程。

结果：Home 和 Pause 两个不同图标执行相同动作。

建议：

- Pause：打开暂停弹窗。
- Home：暂停计时后打开退出关卡确认，确认后回菜单。
- 若不做退出确认，则左侧图标必须改为菜单/更多，而不能继续显示 Home。

### P0-10 9:20 外壳已固定，但内部没有等比缩放

外壳 `#app` 的 9:20 公式正确；内部存在以下破坏等比的硬编码：

- `ui.js:145`：`zOffsetPx = 4`
- `ui.js:151`：牌宽硬上限 `64`
- `ui.js:152`：牌宽硬下限 `36`
- 多处字体使用视口 `vw`，不是 app 容器宽度
- 弹窗星星固定 `56px`
- 多处固定 padding/gap

后果：

- 浏览器放大时，外壳变大，但牌阵到 64px 后停止放大。
- 桌面宽屏、桌面矮窗和手机上的内部比例不一致。

### P0-11 Compact 弹窗通过压扁大面板得到

`modal_panel_large.png` 原始比例为 `940:1480`。`.modal-compact` 将同一张图强制成 `940:1120`，相当于垂直压缩约 24%。

建议：

- 不允许 `background-size:100% 100%` 改变装饰框固有比例。
- 低成本方案：保持大面板原比例并减小整体宽度。
- 正式方案：重新制作 compact 面板，或使用可拉伸的三段式/九宫格框。

### P0-12 关卡选择节点过小，滚动区域利用率低

当前：

- 内层宽度仅占面板约 64%，再分成 5 列并加 `6%` gap。
- 节点实际触控尺寸接近或低于 40px。
- 使用浏览器默认灰色滚动条和上下箭头。
- 面板下半段仍有较大空白。

建议：

- 改为 4 列，节点目标 `44-52` 逻辑 px。
- 网格高度提高到内容区约 `64%-70%`。
- 使用细的深绿/古金滚动条，或改为分页。
- 当前、已完成、锁定三种状态必须在不看小图标时也可区分。

## 4. P1 重要问题

### P1-01 图腾符号的光学尺寸不统一

所有图腾统一使用 `96x96` 槽是正确的，但窄高符号的可见宽度差异很大，例如 spear、comb、mask、water drop。统一 `width:60%; height:50%` 会让部分符号显得明显偏小。

建议保留原文件槽位，通过 metadata 增加少量光学校正：

- `normal`: `1.00`
- `narrow`: `1.10-1.16`
- `very_narrow`: 最大 `1.20`

不得自动裁切并把每张图都撑满。

### P1-02 Settings 中多个开关只有保存值，没有实际效果

源码只写入：

- `sfx`
- `bgm`
- `largeFont`
- `highContrast`

没有找到对应音频、字体或高对比样式应用。尤其 `largeFont` 默认是 `true`，但界面没有变化。

建议：

- 要么实现功能。
- 要么在当前验证版中隐藏未实现项，避免虚假可用性。

### P1-03 Reset Progress 使用浏览器原生 confirm

原生确认框会脱离游戏视觉系统，也可能在 WebView 中表现不一致。应改用同系列确认弹窗。

### P1-04 缺少键盘焦点和状态辅助信息

未发现：

- `:focus-visible`
- toggle 的 `aria-pressed`
- toast 的 `aria-live`
- `prefers-reduced-motion`

同时 `index.html` 禁止缩放。作为纯游戏可以降低优先级，但最终 H5 交付仍应补齐基础焦点和状态表达。

### P1-05 首次加载和低端机性能风险

主要大图：

- `bg_game_1080x2400.png` 约 3.7MB
- `board_stage_bg.png` 约 3.5MB
- `modal_overlay_dark.png` 约 4.3MB
- `modal_panel_large.png` 约 2.6MB

项目没有预加载。移除不透明 modal overlay 的运行时使用后，可立即减少一次大图加载和全屏合成。

### P1-06 未截图状态尚不能验收

当前没有实装截图：

- Settings
- Victory
- Selected tile
- Hint pulse
- Wrong click
- Match animation
- Deadlock toast
- Combo
- Disabled tools
- 关卡滚动后的底部状态

这些只能从源码发现风险，不能据此宣称视觉通过。

## 5. 页面逐屏修改方向

### 主菜单

- 放大品牌主体，而不是单纯放大 `760x420` 透明画布。
- Logo 视觉中心下移到上半屏中部。
- Play 上移，成为第二视觉中心。
- Level/Daily 上移并与 Play 形成稳定组。
- 修复底部叶片粉边。
- 设置按钮保持次要，但触控区不小于 44px。

### 普通游戏页

- 第一优先修复 covered 状态。
- 删除 HUD 重复中心板。
- 底部工具按钮改为图标优先。
- 保持 Home、Level、Timer、Pause 的信息结构。
- 牌阵尺寸随 app 容器等比，不设绝对 `64px` 上限。

### 暂停弹窗

- 背后显示被压暗的当前游戏。
- 修复按钮文字对比。
- 不压扁大面板。
- 三个操作按钮保持同宽、同轴。

### 每日挑战

- 标题改成 `DAILY`。
- 其余复用普通游戏页修复。
- 不另做一套布局。

### 关卡选择

- 4 列优先。
- 放大节点与锁图标。
- 自定义滚动条。
- 让网格占满可用纸面区域。
- 修复所有节点和 close 图标的粉边。

## 6. 源码边界

P0 建议修改：

- `style.css`
- `ui.js`
- `main.js`
- 有问题的 menu/modal/state PNG

P0 不建议修改：

- `engine.js`
- 匹配、撤销、洗牌、提示和可移动判定

条件性修改：

- 只有 covered 状态修复后，牌阵仍显得随机破碎，才评估 `data.js` 的 `trimPositions()`。
- 如要调整，也只调整关卡布局取点的对称性和密度，不改变匹配规则。

## 7. 证据限制

- 当前截图并非严格的自动化同尺寸回归截图，宽高比有轻微裁切差异。
- 早期游戏效果图是概念图，不代表某个确定关卡模板。
- `2.png` 是 Lv.2，不能仅凭牌形轮廓要求与概念图完全相同。
- 可以要求一致的是：信息层级、组件比例、牌体可见性、色板、状态语义和操作结果。
