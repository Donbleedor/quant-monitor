# Mahjong V3 JSON Prefab 可视化编辑器设计规格

日期：2026-06-12

## 1. 目标

在现有原生 HTML/CSS/JavaScript 游戏中增加一套开发专用的 UI 布局编辑能力，使美术或策划不需要直接修改 CSS，也能调整现有组件的：

- 位置
- 宽度和高度
- 锚点
- 层级
- 透明度
- 显示和隐藏
- 字号
- 间距
- 图片资源
- 棋盘安全参数

编辑器的体验类似轻量版 Unity RectTransform / Prefab Inspector，但第一版只编辑现有组件：

- 不新增组件。
- 不删除组件。
- 不改变按钮行为。
- 不改变麻将匹配规则、关卡生成和可解性。
- 不成为独立的第二套页面。

## 2. 已确认的产品决策

| 项目 | 决策 |
| --- | --- |
| 功能范围 | 只编辑现有组件 |
| 启动方式 | `index.html?editor=1` |
| 普通模式 | 不加载编辑器 JS/CSS，不读取编辑器草稿 |
| 逻辑分辨率 | 固定 `360x800`，即 `9:20` |
| 其他尺寸 | `720x1600`、`1080x2400` 自动等比缩放 |
| 编辑器平台 | 仅桌面浏览器 |
| 页面布局 | 左侧 9:20 真实游戏预览，右侧组件树与属性面板 |
| 草稿 | `localStorage` 自动保存 |
| 交付 | 导入/导出 JSON |
| 默认布局 | 程序将确认后的 JSON 放入项目默认配置 |
| 技术栈 | 继续使用原生 HTML/CSS/JavaScript/JSON |

## 3. 方案比较

### 方案 A：为每个组件手写 CSS 变量

每个组件分别定义 `--menu-logo-x`、`--pause-panel-width` 等变量。

优点：

- 运行时稳定。
- 对复杂布局控制精确。

缺点：

- 每增加一个可编辑组件都要改 CSS。
- 变量数量很快失控。
- 编辑器与当前 CSS 强耦合。

### 方案 B：真实 DOM + 通用 RectTransform 适配器

给现有组件加稳定的 `data-ui-id`，编辑器读取真实 DOM 的矩形，通过统一的逻辑坐标转换应用 `width`、`height`、CSS `translate`、字体和样式覆盖。

优点：

- 编辑的就是实际游戏 DOM。
- 不需要复制页面。
- 不需要把全部现有 CSS 重写为编辑器专用布局。
- 能逐步扩大组件覆盖范围。
- 未编辑组件继续走当前 CSS，不会改变默认界面。

缺点：

- 父子组件需要规定应用顺序。
- 少数算法型组件需要专用适配器。
- 必须防止编辑器覆盖现有 `transform` 动画。

### 方案 C：独立编辑画布，导出后再转换成游戏 CSS

优点：

- 编辑器实现相对自由。

缺点：

- 编辑器预览与真实游戏会产生差异。
- 每次 UI 改动都需要同步两套结构。
- 最终效果难以保证。

### 采用方案

采用方案 B。

位置偏移使用 CSS 独立属性 `translate`，不覆盖现有 `transform`、动画和按压效果。普通游戏 DOM、资源和事件绑定仍是唯一事实来源。

## 4. 系统架构

```text
ui-layout.default.json
          |
          v
layout-core.js ------ layout-component-catalog.js
          |                       |
          v                       v
layout-runtime.js ---------- 真实游戏 DOM
          |
          +---- 普通模式：只应用默认 JSON
          |
          +---- editor=1：默认 JSON + localStorage 草稿
                                      |
                                      v
                               layout-editor.js
                               layout-editor.css
```

### 4.1 布局核心

`layout-core.js` 只处理纯数据：

- 配置校验
- 默认配置与草稿合并
- 逻辑坐标和物理坐标转换
- RectTransform 归一化
- 属性白名单
- Undo / Redo 数据历史
- 导入导出格式

该文件不访问 DOM，并同时支持浏览器全局变量与 Node 测试。

### 4.2 组件目录

`layout-component-catalog.js` 定义：

- 页面列表
- 组件 ID
- 显示名称
- DOM selector
- 组件类型
- 父组件
- 可编辑属性
- 资源属性类型
- 应用顺序

导入的 JSON 不允许自定义 selector，避免错误 JSON 选中游戏中任意元素。

### 4.3 布局运行时

`layout-runtime.js` 负责：

- 加载默认 JSON。
- 判断是否为编辑器模式。
- 编辑器模式叠加 localStorage 草稿。
- 在 UI 每次重新渲染后重新应用布局。
- 窗口尺寸变化时重新换算物理像素。
- 默认 JSON 缺失或无效时回退到现有 CSS。

### 4.4 可视化编辑器

`layout-editor.js` 负责：

- 页面预览切换。
- 组件树。
- 点击选择真实组件。
- 选择框和八方向缩放手柄。
- 拖拽、缩放和键盘微调。
- 属性输入面板。
- 锚点选择。
- Undo / Redo。
- 重置组件、页面和全部草稿。
- JSON 导入、导出和校验提示。

## 5. 数据模型

### 5.1 默认布局配置

文件：

`layout/ui-layout.default.json`

格式：

```json
{
  "version": 1,
  "baseRevision": "mahjong-v3-ui-20260612",
  "designSize": {
    "width": 360,
    "height": 800,
    "aspect": "9:20"
  },
  "screens": {
    "menu": {
      "components": {
        "menu.logo": {
          "rect": {
            "x": 180,
            "y": 112,
            "width": 317,
            "height": 248,
            "anchorX": 0.5,
            "anchorY": 0
          },
          "style": {
            "opacity": 1,
            "visible": true,
            "zIndex": 2
          }
        }
      }
    }
  }
}
```

### 5.2 覆盖层原则

现有 CSS 是基础 Prefab。

JSON 只保存覆盖值：

- 没有记录的组件继续使用 CSS。
- 没有记录的属性继续使用 CSS。
- 编辑器首次修改组件时，先捕获当前真实值，再建立覆盖记录。
- 导出 JSON 可以只包含已修改组件。

这样接入编辑器时不需要先手工迁移全部现有布局。

### 5.3 RectTransform

所有坐标使用 `360x800` 逻辑坐标。

```json
{
  "x": 180,
  "y": 432,
  "width": 244,
  "height": 81,
  "anchorX": 0.5,
  "anchorY": 0
}
```

含义：

- `x/y` 是锚点在屏幕中的逻辑坐标。
- `anchorX/anchorY` 取值范围 `0..1`。
- 左上角为 `0,0`。
- 中心为 `0.5,0.5`。
- 右下角为 `1,1`。

逻辑矩形转左上角：

```js
left = x - width * anchorX;
top = y - height * anchorY;
```

物理尺寸：

```js
scale = appRect.width / 360;
physicalLeft = left * scale;
physicalTop = top * scale;
physicalWidth = width * scale;
physicalHeight = height * scale;
```

### 5.4 可编辑样式

第一版白名单：

```json
{
  "style": {
    "opacity": 1,
    "visible": true,
    "zIndex": 2,
    "fontSize": 14,
    "letterSpacing": 1,
    "gap": 8
  }
}
```

不允许 JSON 注入：

- 任意 CSS 字符串
- HTML
- JavaScript
- selector
- event handler

### 5.5 资源覆盖

仅组件目录明确允许的组件可以修改资源。

```json
{
  "asset": {
    "src": "assets/menu/logo_tribal_tiles.png"
  }
}
```

允许扩展名：

- `.png`
- `.webp`
- `.jpg`
- `.jpeg`
- `.gif`

路径必须是项目相对路径，禁止：

- `javascript:`
- `data:`
- `http://`
- `https://`
- `..`
- 绝对磁盘路径

## 6. 组件类型

### 6.1 Rect 组件

支持：

- x/y
- width/height
- anchor
- zIndex
- opacity
- visible

适用：

- Logo
- 按钮
- HUD
- Modal
- 工具栏
- 图片装饰

### 6.2 Text 组件

在 Rect 基础上增加：

- fontSize
- letterSpacing

不允许编辑文字内容，避免编辑器承担本地化系统职责。

### 6.3 Group 组件

用于：

- 菜单二级按钮组
- Modal 按钮组
- 统计区域
- Settings 列表

支持：

- rect
- gap
- opacity
- visible

### 6.4 Family 组件

一个配置同时作用于重复元素：

- 关卡节点
- 星星
- 统计卡

点击任意实例时选择整个 Family，不保存单个动态实例的位置。

### 6.5 Board 组件

棋盘属于算法布局，不能逐张牌拖动。

可编辑：

- boardArea rect
- tileMaxWidth
- zOffset
- totemWidthPercent
- totemHeightPercent
- boardPadding
- coveredOpacity

禁止：

- 单张牌 x/y
- typeId
- 层叠关系
- 牌数
- 匹配逻辑

## 8. DOM 标记规则

所有可编辑组件增加稳定标记：

```html
<div class="menu-logo-wrap" data-ui-id="menu.logo">
```

规则：

- `data-ui-id` 必须全局稳定。
- ID 与组件目录一致。
- 动态 Family 的多个实例可以使用相同 ID。
- 业务事件继续使用现有 `id`，不替换为 `data-ui-id`。
- 编辑器只使用 `data-ui-id`，不依赖易变化的 class 层级。

## 9. RectTransform 应用算法

### 9.1 捕获

```js
const appRect = app.getBoundingClientRect();
const rect = element.getBoundingClientRect();
const scale = appRect.width / 360;

const left = (rect.left - appRect.left) / scale;
const top = (rect.top - appRect.top) / scale;
const width = rect.width / scale;
const height = rect.height / scale;

x = left + width * anchorX;
y = top + height * anchorY;
```

### 9.2 应用

1. 清除该元素之前的 editor `translate`。
2. 应用 width/height。
3. 重新测量元素。
4. 计算目标左上角与当前左上角差值。
5. 使用 CSS 独立 `translate` 属性移动。
6. 应用 opacity、visibility、zIndex 和文字样式。

禁止覆盖 `transform`，避免破坏：

- Logo 浮动
- 按钮按压
- Tile 动画
- Modal 动画

### 9.3 父子顺序

组件目录定义 `parentId` 和 `depth`。

运行时：

- 先应用父组件。
- 再应用子组件。
- Family 最后应用。

如果父组件移动、子组件没有覆盖值，子组件跟随父组件。

如果父子组件都有覆盖值，子组件最终回到 JSON 记录的全局逻辑坐标。

## 10. 编辑器界面

### 10.1 桌面布局

```text
+------------------------------+------------------+
|                              | Screen selector  |
|      360x800 Game Preview    | Mode             |
|                              | Component tree   |
|      Selection overlay       | Inspector        |
|                              | History          |
|                              | Import / Export  |
+------------------------------+------------------+
```

### 10.2 顶部工具栏

- Screen
- Inspect / Interact
- Undo
- Redo
- Reset Component
- Reset Screen
- Reset All Draft
- Import JSON
- Export JSON

### 10.3 Inspect 模式

- 捕获预览内点击。
- 阻止游戏按钮业务事件。
- 选择最内层有 `data-ui-id` 的元素。
- 显示选择框。
- 可拖动和缩放。

### 10.4 Interact 模式

- 不拦截游戏点击。
- 用于检查按钮、弹窗和页面流程。
- 保留当前选中组件，但隐藏缩放手柄。

### 10.5 属性面板

Rect：

- X
- Y
- Width
- Height
- Anchor 3x3
- Lock aspect ratio

Style：

- Z Index
- Opacity
- Visible
- Font Size
- Letter Spacing
- Gap

Asset：

- 当前相对路径
- 选择项目内路径文本
- Restore default

### 10.6 操作细节

- 拖动：实时更新 x/y。
- 八个缩放手柄：更新 rect。
- 方向键：移动 1 逻辑像素。
- Shift + 方向键：移动 10 逻辑像素。
- Shift + 拖动：锁定水平或垂直轴。
- Aspect Lock：修改宽度时同步高度。
- 输入框 Enter：提交一条历史记录。
- Esc：取消当前拖动。

## 11. 页面预览控制器

页面选择器不复制 UI，直接调用现有渲染逻辑。

| Screen | 预览方法 |
| --- | --- |
| Menu | `Game.showMenu()` |
| Gameplay | `Game.startLevel(1)` 后停止计时 |
| Daily | `Game.playDailyChallenge()` 后停止计时 |
| Pause | 渲染 Gameplay，再调用 `UI.showPauseModal()` |
| Victory | 渲染 Gameplay，再调用 `UI.showWinModal()` 测试数据 |
| Levels | 渲染 Menu，再调用 `UI.showLevelSelectModal()` |
| Settings | 渲染 Menu，再调用 `UI.showSettingsModal()`，使用 settings 副本 |
| Reset | 渲染 Settings，再触发 `#modalClearData` |

预览回调必须使用 no-op，禁止：

- 清除真实存档
- 解锁关卡
- 写入成绩
- 自动进入下一关

## 12. 草稿、历史与导入导出

### 12.1 localStorage

草稿键：

`tribalTiles.uiLayoutDraft.v1`

编辑器偏好键：

`tribalTiles.uiEditorPrefs.v1`

普通模式不读取以上两个键。

### 12.2 自动保存

- 修改后 200ms 防抖保存。
- 页面关闭前同步保存最后状态。
- 面板显示 `Saved`、`Unsaved` 或 `Invalid`。

### 12.3 Undo / Redo

- 内存中保存最近 100 条。
- 一次完整拖拽只产生一条记录。
- 连续输入同一字段在 400ms 内合并。
- 导入、Reset Screen、Reset All 各产生一条记录。

### 12.4 导出

文件名：

`tribal-tiles-layout-v1.json`

导出内容：

- version
- baseRevision
- designSize
- screens
- generatedAt

不导出：

- 编辑器 UI 偏好
- 当前选择
- Undo 历史
- 浏览器绝对路径

### 12.5 导入

流程：

1. 读取 JSON。
2. 校验 schema。
3. 显示错误列表。
4. 校验通过后覆盖编辑器草稿。
5. 重新应用当前页面。
6. 允许 Undo 恢复导入前状态。

## 13. 普通模式隔离

普通 URL：

`index.html`

普通模式：

- 加载布局核心、组件目录和运行时。
- 加载默认 JSON。
- 不加载 `layout-editor.js`。
- 不加载 `layout-editor.css`。
- 不读取 localStorage 草稿。
- 不创建编辑器 DOM。
- 不拦截业务点击。

编辑 URL：

`index.html?editor=1`

编辑器资源由 `layout-runtime.js` 动态注入。

## 14. 移动端门禁

当满足任一条件：

- viewport 宽度小于 900px
- viewport 高度小于 700px
- 精细指针不可用

编辑器显示：

`UI Editor requires a desktop browser.`

移动端门禁不影响普通游戏。

## 15. 错误处理

### 默认 JSON 加载失败

- 控制台输出一条 warning。
- 继续使用当前 CSS。
- 游戏正常启动。

### 默认 JSON 校验失败

- 忽略无效配置。
- 输出具体路径，例如：

`screens.menu.components.menu.logo.rect.width must be > 0`

### DOM 组件暂时不存在

- 跳过该组件。
- 在 editor 状态栏显示 missing 数量。
- 页面重新渲染后再次尝试。

### 导入未知组件

- 拒绝未知组件 ID。
- 不允许 JSON 指定 selector。

### 图片资源加载失败

- 保留原资源。
- 属性面板显示错误。
- 不将失败路径保存到草稿。

## 16. 性能

- 普通模式不运行 MutationObserver 扫描全页面。
- 只在 UI 明确 mount 后应用当前页面。
- resize 使用 requestAnimationFrame 合并。
- 拖拽每帧最多应用一次。
- localStorage 写入防抖。
- Editor DOM 只在 `editor=1` 创建。

## 17. 测试与验收

### 核心单元测试

- 配置校验
- 默认与草稿合并
- RectTransform 坐标转换
- 属性白名单
- 资源路径校验
- Undo / Redo
- schema version 拒绝

### 普通模式

- 编辑器 JS/CSS 未加载。
- `window.UIEditor` 不存在。
- localStorage 草稿不影响游戏。
- 360x800 与 720x1600 外观不回退。

### 编辑模式

- 预览为 360x800。
- 页面切换可用。
- Inspect 不触发业务点击。
- Interact 可操作游戏。
- 拖拽、缩放、锚点和输入框一致。
- 刷新后草稿恢复。
- 导入导出往返一致。
- Reset Component / Screen / All 正确。
- 移动端门禁正确。

### 硬性失败条件

- 普通模式加载编辑器资源。
- 编辑器草稿污染普通模式。
- 修改布局导致玩法引擎变化。
- 单张麻将牌可以被任意拖动。
- 导入 JSON 可以执行任意 CSS、HTML 或脚本。
- 页面重新渲染后布局覆盖丢失。
- 720x1600 不再是 360x800 的等比结果。
- 导出的 JSON 无法重新导入得到相同布局。

## 18. 非目标

第一版不做：

- 新增组件
- 删除组件
- 复制组件
- 自由分组
